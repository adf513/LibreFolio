# F-051 — Transaction AssetEvent Link

## What it does
A foreign key relationship between `Transaction` and `AssetEvent` that enables DIVIDEND/INTEREST transactions to reference the global asset event they derive from. The link supports auto-suggestion (Staging Modal) and is intentionally nullable so destructive flows (see [[decisions/policy-d-currency-wipe]]) can sever the link without losing the transaction row.

## Layer breakdown
**Backend**: `backend/app/db/models.py` — `Transaction.asset_event_id` (nullable FK to `AssetEvent.id`, index `idx_transactions_asset_event`); `backend/app/api/v1/transactions.py` — `POST /transactions/events/suggest` endpoint; `backend/app/alembic/versions/001_initial.py` — single-migration column.

## Dependencies
- [[F-046]] Transaction Model — the FK lives on Transaction
- [[F-031]] Asset Events — the target of the FK

## Enables
- [[decisions/policy-d-currency-wipe]] — Policy D wipe sets `asset_event_id = NULL` instead of deleting the transaction row, preserving FIFO history.
- (future) smart assistant for retroactive event matching

## Implementation notes
- Phase 7 Part 1: `asset_event_id` column added to `Transaction` (nullable FK to `AssetEvent.id`, with `idx_transactions_asset_event` index).
- **Phase 7 Part 3**: `POST /transactions/events/suggest` endpoint implemented:
  - Input: batch of transaction stubs (asset_id, date, type, amount)
  - Output: per-TX list of candidate AssetEvents within ±`tolerance_days` (0–7), ordered by date distance
  - Used by the Staging Modal for auto-linking DIVIDEND/INTEREST transactions to events
- The suggest endpoint preserves request order in the response.
- Tolerance slider (0–7 days): `tolerance_days=0` requires exact date match.
- Link is optional (nullable FK): manually entered transactions don't require an event link.
- Type mapping: DIVIDEND TX → DIVIDEND event; INTEREST TX → INTEREST event; others → SPLIT/PRICE events.
