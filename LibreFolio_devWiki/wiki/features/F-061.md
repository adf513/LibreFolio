# F-061 — 5-layer Provider Cache

## What it does
A 5-layer TTL cache system for asset provider responses to avoid redundant network calls and rate-limit violations. Each cache is scoped to a specific use case with an appropriate TTL. All caches use `theine` timer wheel for automatic TTL eviction.

| Cache | TTL | Scope |
|-------|-----|-------|
| `asset_history_fetch` | 15 min | Smart date-range per asset — fetches only missing gaps |
| `asset_current_fetch` | 2 min | Current price per asset — supports frontend polling without hammering provider |
| `asset_metadata_fetch` | 30 min | Provider metadata per asset — for `fetch_asset_metadata()` |
| `search_queries` | 15 min | Exact search query deduplication |
| `search_results` | 24h | Individual search result items |

## Layer breakdown
**Backend**: `backend/app/services/asset_source.py` (`_run_provider_in_thread()` calls wrapped with caches), `backend/app/utils/cache_utils.py` (`get_ttl_cache()` helper using `theine`)

FX also has its own cache: `_fx_fetch_cache` (TTL 5min) in `backend/app/services/fx.py`.

## Dependencies
- [[F-025]] Asset Provider Registry — provider calls that are cached
- [[F-060]] Thread Isolation — cache wraps thread-isolated calls

## Enables
- [[F-027]] Asset Data Sync — cache prevents repeated provider calls during bulk sync

## Implementation notes
- `probe_provider_config()` bypasses ALL caches — always a live call (dry-run test)
- Cache key: `(provider_code, identifier, date_range_hash)` for history; `(provider_code, identifier)` for current
- `theine` TTL cache: timer wheel eviction — zero GC pressure from time-based entries
- Cache is per-process (no Redis) — suitable for single-instance self-hosted deployment
