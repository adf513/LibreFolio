# F-056 — FIFO at Runtime

## What it does
Computes the cost basis of asset holdings using First-In-First-Out matching, on demand at query time. The result is **never persisted** — it is recalculated every time a portfolio value or gain/loss is requested.

## Layer breakdown
**Backend only**: the FIFO engine runs in `backend/app/services/transaction_service.py`. The frontend only receives the computed results (gains, cost basis per lot) via API responses.

## Dependencies
- [[F-046]] Transaction Model — needs complete, ordered transaction history per asset
- [[F-030]] Price History — needs current/historical prices for market value calculation

## Enables
- [[F-054]] Dashboard KPI — accurate P&L requires FIFO cost basis
- [[F-058]] ROI Calculations — FIFO determines realized/unrealized gains

## Implementation notes
- FIFO matching is O(N) per asset per query — iterates transactions chronologically, maintains open lots
- **Not persisted**: avoids stale data problems when transactions are edited retroactively
- Consequence: editing old transactions (e.g. fixing a BRIM import error) immediately corrects all derived values
- At Phase 7 launch: only FIFO supported. LIFO/PMC deferred to Phase 8+

## Dead ends / Ideas explored
- **Persisting FIFO lots** (considered in V1): rejected because retroactive edits would require invalidation cascades across potentially hundreds of cached values. Runtime recalculation is simpler and fast enough at current scale.
