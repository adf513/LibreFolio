# F-054 — Dashboard KPI & Overview

## What it does
The main Dashboard Home at `/dashboard` showing portfolio-level KPI cards (Net Worth, Gain/Loss, Weighted ROI —
TWRR + MWRR), a broker filter + date-range picker, GrowthChart, Allocation panel, the Holdings/Performance
positions panel (see [[concepts/holdings-performance-panel]]), and a Recent Transactions widget. All data comes
from a single `POST /portfolio/report` call (see [[concepts/portfolio-report-unified]]) — no separate KPI
endpoint. NAV and P&L are aggregated across the user's visible/selected brokers by the
[[entities/portfolio-engine]], not by naive per-broker `share_percentage` averaging.

## Layer breakdown
**Backend**: `POST /portfolio/report` (`backend/app/api/v1/portfolio_api.py`) → `PortfolioService.get_report()` → [[entities/portfolio-engine]]. KPIs (Net Worth / Period P&L / TWRR / MWRR) computed via `roi_utils.py`.

**Frontend**: `frontend/src/routes/(app)/dashboard/+page.svelte` — `KpiSection.svelte`, `GrowthChart.svelte`, `AllocationPanel.svelte`, `PositionsPanel.svelte` (Holdings/Performance tabs), `DataQualityBanner`, AI export to clipboard.

## Implementation notes
- KPI row: Net Worth | Gain/Loss | Weighted ROI (TWRR + MWRR), sourced from the engine's NAV/pool computation — see [[concepts/3-pool-cash-model]]
- `DataQualityBanner` reads the unified `data_quality` field (populated by `build_data_quality_report()`, wired 2026-07-06)
- Date range shared with assets/fx pages via `dateRangeController.svelte.ts`
- AI portfolio export to clipboard (`aiExportClipboard.ts`) — added alongside the Holdings/Performance refactor; reworked 2026-07-15 into a 6-entry single-purpose prompt catalog, see [[decisions/ai-export-prompt-catalog]]
- Phase 9 M1 (backend foundations) + M2 (dashboard + engine) — archived 2026-07-07, see [[sources/phase09-m1-m2-archive-2026-07]]

## Dead ends / Ideas explored
- A separate `GET /api/v1/portfolio/overview` endpoint (originally planned) was never built — superseded early by the unified `/portfolio/report` design.
- A separate `/allocation-history` endpoint was also considered and rejected — see [[decisions/portfolio-summary-direct-wiring]].
