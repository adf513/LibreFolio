# F-048 — Staging Modal

## What it does
A unified modal for staging transactions before atomic commit. Currently supports **manual-only** mode (`create-many`, `edit-many`). Planned future mode: `create-brim` (BRIM import output → staging). Features (manual mode): asset-grouped rows with broker color coding, AssetSelect for per-row asset assignment, amount + fee + event tolerance slider (0–7 days), validate-only dry-run preview, and atomic multi-broker commit via `POST /transactions/bulk`.

## Implementation notes — Manual Mode (Part 4, done)

- **`create-many` mode**: empty draft rows for manual entry; user adds per-row asset, type, date, quantity, price, fee
- **`edit-many` mode**: pre-fills from selected existing transactions; user edits and re-validates
- **`POST /transactions/validate`** called with exactly ONE populated list (`creates`, `updates`, or `deletes`) — never mixed; see [[decisions/multi-broker-atomic-tx]]
- **Draft DraftRow[]**: rows held in `$state(drafts)`. Computed locally first → single `drafts = next` write (fixes W1 infinite $effect loop; see [[problems/svelte5-effect-read-write-loop]])
- **AssetSelect.svelte** used for per-row lookup
- **Atomicity**: single `POST /transactions/bulk` — all succeed or all fail (multi-broker supported)
- **Broker note**: `Import ▾` button deferred (needs brokerId picker added in Part 5)

## Implementation notes — BRIM Mode (Part 5, planned)

- Unknown assets (from BRIM fake IDs) matched to real assets via `AssetMatchingWizard` or `AssetModal` create flow
- Tolerance slider (0–7 days): matches BRIM dividend transactions to existing `AssetEvent` records within tolerance window
- One modal, multiple source modes — `create-brim` mode arriving in Phase 7 Part 5
