# F-060 — Thread Isolation for Providers

## What it does
All asset provider calls are executed in dedicated threads with isolated event loops via `_run_provider_in_thread()` in `backend/app/services/asset_source.py`. This allows providers to use synchronous I/O libraries (like `yfinance`, `httpx` sync, `requests`) directly without `asyncio.to_thread()` gymnastics — each thread gets its own event loop. Prevents thread-safety issues between concurrent provider calls.

## Layer breakdown
**Backend**: `backend/app/services/asset_source.py` — `_run_provider_in_thread()` utility, used by `AssetSourceManager` for all sync/fetch operations

## Dependencies
- [[F-025]] Asset Provider Registry — providers that benefit from isolation
- [[F-059]] Provider Registry Pattern — discovered providers called via this mechanism

## Enables
- [[F-027]] Asset Data Sync — sync uses thread-isolated provider calls
- [[F-061]] 5-layer Provider Cache — cache wraps the thread-isolated calls

## Implementation notes
- Each provider call: `ThreadPoolExecutor(max_workers=1)` per call, fresh event loop inside
- Providers can use sync I/O directly (no `asyncio` needed inside provider classes)
- Parallel calls: `asyncio.gather()` with multiple thread-isolated calls for "sync all"
- `probe_provider_config()` bypasses cache (dry-run, always fresh)
- Same isolation pattern used for FX providers and asset providers
